# 手机端菜单修复设计

## 问题结论

手机端汉堡菜单的同一次点击被 `js/nav.js` 和 `js/main.js` 分别处理。`nav.js` 先打开菜单，事件继续冒泡后 `main.js` 又将其关闭，因此用户看到“点击没反应”。

微信截图中的 `ERR_PROXY_CONNECTION_FAILED` 已通过另一部手机对照测试确认是单机微信代理状态问题：同一网站可在另一部手机微信中打开，网站 HTTPS、HTTP 跳转和模拟微信 User-Agent 响应均正常。本次不调整域名、证书或托管架构。

## 修复方式

- `js/nav.js` 继续负责注入导航并绑定汉堡菜单点击。
- 删除 `js/main.js` 中重复的汉堡菜单事件委托。
- 保留 `js/main.js` 的搜索和详情展开功能。
- 不改变导航结构、样式、主题选择器或内容。

## 测试

- 增加回归测试，断言汉堡菜单监听只由 `js/nav.js` 持有，`js/main.js` 不再包含 `.hamburger` 点击处理。
- 保留既有主题、敏感内容删除和栏目壳测试。
- 全量测试通过，`git diff --check` 无错误。
