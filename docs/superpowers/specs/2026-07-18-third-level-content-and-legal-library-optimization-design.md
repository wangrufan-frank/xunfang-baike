# 三级内容、公开来源与法律法规库优化设计

- 日期：2026-07-18
- 状态：待实施
- 交付对象：Claude Code
- 适用范围：桌面端与移动端网站；本期不改微信小程序

## 1. 背景与目标

当前网站三级内容页采用连续长文结构，典型页面由一段摘要和 8～10 个篇幅相近的章节顺序堆叠。信息虽然完整，但首屏没有明确的行动结论、风险边界和学习路径，页面整体更像论文，用户难以在短时间内抓住重点。

现有资料来源还直接展示内部知识库文件名及“巡防百科知识库内部资料”字样。内部资料可以继续作为内容生产、溯源和维护依据，但不应进入公开页面。

执法规范模块还有两项内容缺口：

1. “执法程序与法条应用”页面只概括法律依据或引导用户跳转，未把实际适用的具体条款完整展示在当前页面。
2. “法律法规库”页面主要是法规介绍、少量核心条款和官方链接，不是可搜索、可定位、可复制的完整法规正文。

本次改造要实现以下结果：

- 用户进入三级页后，在首屏 10 秒内看懂“这是什么、先做什么、不能做什么、依据是什么”。
- 次要说明按需展开，降低首次阅读负担，但内容仍可访问、可搜索、可打印。
- 公开页面只展示互联网来源；内部来源只保留在维护数据中。
- 执法规范页就地展示所引用法条的完整原文。
- 法律法规库站内收录当前有效版本的完整正文，不再以简介和外链代替正文。

## 2. 现状基线

实现前应以当前主工作区重新统计一次，以下为 2026-07-18 的审计基线：

- 六个内容模块共有 117 个 HTML 文件（含模块首页），其中 83 个页面展示了“巡防百科知识库内部资料”。
- 87 个页面使用静态 `.source-list`；内部来源与互联网来源混排。
- `data/content-inventory.json` 同时记录 `source_files` 和 `public_sources`，前者包含本地绝对路径。
- `data/public-sources.json` 与 `tools/public_source_index.py` 已具备公开来源核验和知识点引用能力，但当前生产 HTML 未渲染 `public-source-index` 标记，且账本只覆盖部分内容页。
- `fagui/index.html` 下有 5 个“执法程序与法条应用”页面、7 个“法律法规库”入口。
- 现有法律库页面结构通常为“法规概述—版本与效力—核心条款—巡防关联—注意事项—官方全文入口”，正文不完整。
- 当前 CSS 对三级页主要只有摘要宽度、章节间距和来源卡片样式，没有首屏速览、章节导航、重点卡片或法规正文的专用组件。

## 3. 方案选择

### 方案 A：只做视觉压缩

保留全部章节常显，仅缩短摘要、减小段间距并增加侧边目录。

优点是改动小；缺点是内容密度没有实质下降，仍然是“长论文”，不能满足“先重点、后细节”的核心需求。

### 方案 B：按内容类型实施渐进式披露（采用）

普通知识页和执法应用页采用“首屏速览 + 重点常显 + 细节按需展开”；法律法规全文页采用“法规状态卡 + 章条目录 + 完整正文常显”。两类页面共享视觉语言，但不强行使用同一种折叠规则。

优点是既降低普通页面认知负担，又保证法规正文完整、可搜索和可引用；缺点是需要建立清晰的内容模板和数据校验。

### 方案 C：将长内容拆成更多独立页面

每个章节拆成一页，通过学习路径串联。

优点是单页短；缺点是页面数量、维护成本和跳转成本大幅上升，移动端学习连续性变差，也容易造成搜索结果碎片化。

结论：采用方案 B。它最贴合现有静态站点架构，不需要引入前端框架，也不会牺牲法规全文的完整性。

## 4. 信息架构

三级页分为两种模板，不得继续用同一个长文模板处理所有内容。

### 4.1 学习型三级页

适用于：`jingqing/`、`qinwu/`、`xunlian/`、`zhuangbei/`、`zoufang/` 的详情页，以及 `fagui/` 下“现场规范用语”“执法程序与法条应用”页面。

从上到下的固定结构为：

1. 面包屑。
2. 标题区：标题、一句话定位、内容类型标签、预计阅读时间。
3. 首屏速览区。
4. 页内导航。
5. 重点正文区。
6. 深入学习区。
7. 具体法条区（存在法律依据时）。
8. 互联网资料来源。
9. 相关内容与上一篇/下一篇。

### 4.2 法规全文页

适用于 `fagui/index.html#legal-library` 下每一部单独的法律、行政法规、部门规章或公开规范性文件。

从上到下的固定结构为：

1. 面包屑。
2. 法规标题与效力状态。
3. 版本元数据卡。
4. “巡防常用条款”快捷入口（只作导航，不替代全文）。
5. 章条目录。
6. 完整法规正文。
7. 官方互联网来源与核验信息。
8. 关联执法应用页面。

法规全文页不使用 `<details>` 隐藏正文。所有条文默认常显并存在于初始 HTML 中，确保浏览器页内查找、站内搜索、复制、打印和无脚本访问均可用。

## 5. 学习型三级页详细设计

### 5.1 首屏速览

新增 `.quick-read`，桌面端为 2×2 卡片，移动端为单列。卡片数量控制在 3～4 张，不得为了统一而填充空洞内容。

建议字段：

- `核心结论`：一句话说明本页最重要的判断。
- `先做什么`：3～5 个短步骤，每项只保留一个动作。
- `不能做什么`：最关键的 2～4 项禁止行为，使用明确否定句。
- `法律依据 / 风险提示`：有法律依据时显示法规名称和条款号；无直接法条时显示最重要的风险边界。

速览内容必须由正文提炼，不得另造与正文不一致的新结论。每张卡片可链接到正文对应锚点。

### 5.2 正文分层

正文按重要性分为两层：

- `重点正文` 默认展开：适用边界、现场流程、关键判断、禁止事项、核心法律依据。
- `深入学习` 默认折叠：背景解释、完整风险分析、协同细节、记录要求、扩展案例等。

深入学习使用原生 `<details class="learning-section">` 与 `<summary>`，不得用只靠 JavaScript 的自制折叠组件。页面需提供“展开全部 / 收起全部”控制；无 JavaScript 时每个章节仍可独立展开。

以下内容不得默认折叠：

- 可能影响人身安全的警告；
- 明确禁止行为；
- 执法授权、适用条件和程序前置条件；
- 直接影响权利义务的时间、范围、审批和告知要求；
- 法条原文。

### 5.3 内容表现组件

将长段落按语义转换为少量稳定组件，避免“每段都卡片化”造成新的视觉噪声：

- `.action-steps`：有顺序的处置或操作步骤；使用有序列表。
- `.do-dont-grid`：应当 / 不得对照。
- `.key-warning`：高风险或禁止性信息；每节最多一个。
- `.fact-list`：条件、对象、时限、记录要求等短事实。
- `.legal-basis-card`：法规名、条款号、完整条文、适用解释。
- `.article-toc`：页内章节导航，桌面端可在正文侧边粘滞，移动端显示为顶部可横向滚动的锚点列表或下拉导航。

正文最大阅读宽度建议为 720～760px。桌面端总容器可加宽，以容纳正文和 240～280px 的目录栏；移动端取消双栏。所有组件继续使用 `css/style.css` 中现有主题变量，不能引入第二套颜色系统。

### 5.4 折叠与链接行为

- 点击页内目录定位到折叠章节时，脚本应先展开对应 `<details>`，再滚动并转移焦点。
- URL 带章节 hash 时，页面加载后自动展开对应章节。
- 浏览器“打印”样式必须展开全部章节，并隐藏展开按钮、粘滞目录和纯导航元素。
- 站内搜索索引继续包含折叠内容；搜索结果跳转到章节锚点时自动展开。
- 不保存跨页面的折叠状态，避免用户再次进入页面时看不到默认重点结构。

## 6. 公开来源设计

### 6.1 展示规则

公开页面的“资料来源”只允许出现可访问的互联网来源。页面中不得出现：

- 内部 Markdown 文件名；
- “巡防百科知识库内部资料”等内部来源标签；
- `F:\...` 等本地绝对路径；
- 仅有名称、没有互联网链接的来源项。

每个公开来源至少展示：标题、发布机构或平台、可点击 URL、核验日期。法律法规来源还必须展示版本或效力状态。

内部来源不删除，继续保留在 `data/content-inventory.json` 的 `source_files` 中，供内容维护和内部追溯使用；渲染器和页面模板必须明确忽略该字段。

### 6.2 单一公开来源账本

`data/public-sources.json` 应成为公开展示的唯一数据源，`data/content-inventory.json.public_sources` 只作为迁移输入和覆盖审计依据，不再直接拼接 HTML。

实施时将公开来源账本扩展到全部三级内容页：

- 每页至少建立页面级公开来源列表。
- 已有知识点级引用的数据继续保留，可在具体段落后显示引用编号。
- 暂时不能建立知识点级映射的页面，先显示页面级来源，但不得伪造精确对应关系。
- 无可核验互联网来源的页面不得生成空的“资料来源”卡片；应在构建检查中报错并阻止发布，而不是回退展示内部资料。

最终页面只保留一个来源组件，避免现有 `.source-list` 与未来 `.public-source-index` 重复。建议统一为 `<section class="source-list source-list--public">`，由 `tools/public_source_index.py` 生成；知识点引用锚点继续沿用现有稳定 ID 规则。

### 6.3 来源优先级

法律内容按以下顺序选源：

1. 国家法律法规数据库、国家行政法规库；
2. 全国人大、中国政府网、司法部、公安部；
3. 发布机关的省级或市级政府/公安机关官网；
4. 只有在无上述来源时才使用其他可核验公开网站，并在维护数据中说明原因。

普通知识内容不强制全部来自政府网站，但必须使用公开可访问、标题与发布机构明确的互联网页面。

## 7. 执法程序页面的具体法条设计

“执法程序与法条应用”5 个页面必须在相关论述之后就地展示 `.legal-basis-card`，不能只在末尾提供“条文原文入口”。每张法条卡包含：

- 法规全称；
- 条款号；
- 该条完整原文，包括该条下的全部款、项、但书和时限；
- 当前效力状态与版本日期；
- “本页如何适用”简短解释，明确区分原文与编辑说明；
- 指向站内法规全文相应条款的深链接；
- 官方互联网来源链接和核验日期。

“详细展示具体条款”的判定标准是：用户不离开当前执法程序页，就能读到支撑该项操作的完整条文。不能只摘录半句、只写条款要点，也不能用“参见某法第九条”代替原文。

如果同一条文在多个应用页复用，HTML 必须由统一法规数据生成，禁止人工复制多份后分别维护。应用页可以选择引用哪些条，但条文文本必须与法规库中的同版本原文逐字一致。

## 8. 法律法规库完整正文设计

### 8.1 完整性的定义

每个法规全文页必须包含：

- 正式标题；
- 发布机关、公布日期、施行日期、文号、效力状态和当前版本说明；
- 题注或修订沿革；
- 目录（原文有目录时）；
- 从第一条到最后一条的全部现行条文；
- 原文中的章、节、款、项层级；
- 附则和属于法规正文的附件；
- 官方来源链接与本站核验日期。

编辑性“法规概述”“巡防关联”“注意事项”可以保留在全文之前的导读区，但必须标记为“本站导读”，且不能插入法规原文内部。原文区域标题固定为“法规全文”。

### 8.2 当前 7 个入口的处理

- `zhian-guanli-chufa-fa.html`：替换为 2025-06-27 公布、2026-01-01 施行的现行《中华人民共和国治安管理处罚法》完整文本。现有旧版摘要不得继续作为现行条款依据。
- `renmin-jingcha-fa.html`：收录现行《中华人民共和国人民警察法》完整文本。
- `jumin-shenfenzheng-fa.html`：收录现行《中华人民共和国居民身份证法》完整文本。
- `xingzheng-anji-chengxu-guiding.html`：收录现行有效、包含后续修正的《公安机关办理行政案件程序规定》完整文本，不得继续使用 2012 年初始版本代替现行版本。
- `jingxie-wuqi-tiaoli.html`：收录现行《中华人民共和国人民警察使用警械和武器条例》完整文本。
- `xianchang-zhizhi-guicheng.html`：以公开发布机关页面核验后，收录《公安机关人民警察现场制止违法犯罪行为操作规程》完整文本。
- `qita-xiangguan-guifan.html`：该页当前不是单一法规，不应伪装成“完整法规正文”。将其改为“其他规范索引”，每部公开且现行的规范建立独立全文页；不能获得权威公开全文或不能确认现行效力的文件只保留在研究清单，不进入法律法规库。

### 8.3 法规正文交互

- 章标题使用稳定锚点，如 `#chapter-2`；条文使用 `#article-9`。
- 每条旁提供“复制条文链接”，不提供默认折叠。
- 目录突出当前阅读位置；点击条款号可直接定位。
- 支持浏览器页内查找“第九条”等精确查询。
- 移动端条文编号独占或固定在段首，不采用过窄双栏。
- 打印时隐藏网站导航，只输出法规元数据、完整正文、官方来源和打印日期。

## 9. 法规数据模型与生成方式

新增 `data/legal-documents.json` 作为法规正文唯一事实源，避免在 7 个以上 HTML 文件和 5 个执法应用页中重复维护条文。建议结构如下：

```json
{
  "version": 1,
  "documents": [
    {
      "id": "renmin-jingcha-fa",
      "title": "中华人民共和国人民警察法",
      "document_type": "法律",
      "authority": "全国人民代表大会常务委员会",
      "document_number": "",
      "status": "现行有效",
      "promulgated_at": "",
      "effective_at": "",
      "version_label": "",
      "source_url": "https://...",
      "verified_at": "2026-07-18",
      "history": [],
      "chapters": [
        {
          "number": "第一章",
          "title": "总则",
          "articles": [
            {
              "number": 1,
              "label": "第一条",
              "paragraphs": ["完整原文第一款", "完整原文第二款"]
            }
          ]
        }
      ]
    }
  ]
}
```

新增或扩展构建工具，从该数据生成：

- 法规全文 HTML；
- 执法应用页中的法条卡；
- 法规目录与条文锚点；
- 搜索索引中的法规标题、条款号和正文；
- 完整性测试所需的条文数量与首末条信息。

不得让构建脚本联网抓取后直接发布。法规文本的引入应是“人工选择官方来源—保存结构化文本—双人或至少二次核对—本地构建”。网络抓取只能作为导入辅助，不能替代版本和完整性核验。

## 10. 文件级改造边界

Claude Code 实施时预计修改或新增：

- `css/style.css`：速览、页内目录、折叠章节、步骤、警告、法条卡、法规元数据和法规正文样式。

- 六个模块中的三级 HTML：迁移到学习型模板；不改变现有 URL。
- `fagui/index.html`：将法律法规库说明改为“站内完整现行正文”，处理“其他规范索引”。
- `fagui/*.html`：5 个执法应用页加入具体法条，7 个现有法规入口迁移为全文或索引。
- `data/legal-documents.json`：新增法规结构化正文。
- `data/public-sources.json`：扩展到全部三级内容页并作为公开展示唯一账本。
- `data/content-inventory.json`：保留内部溯源字段；必要时增加模板类型和法规 ID，但不得删除维护所需的 `source_files`。
- `tools/public_source_index.py`：只渲染互联网来源、覆盖所有三级页、移除重复来源区。
- 新增 `tools/build_legal_pages.py` 或在现有内容构建工具中增加独立法规生成模块。
- `tools/build_search_index.py`：索引折叠正文和完整法规条文，并支持条款锚点。
- `tests/`：增加本文第 12 节所列的结构、内容、来源和法规完整性测试。

不在本期范围内：重做一级首页、重做二级模块首页整体视觉、修改登录鉴权、调整微信小程序、重写内部知识库内容。

## 11. 数据流与失败处理

### 11.1 内容生成

```text
内部资料（仅维护） ──> content-inventory.source_files ──> 不进入公开 HTML
互联网来源 ─────────> public-sources.json ────────────> 页面公开来源区
官方法规原文 ───────> legal-documents.json ──┬───────> 法规全文页
                                             ├───────> 执法页法条卡
                                             └───────> 搜索索引
```

### 11.2 构建失败条件

以下任一情况必须使检查失败并给出具体文件和字段，不能静默跳过：

- 页面公开来源中出现内部文件名、本地路径或无 URL 项；
- 法规页面缺少官方来源、效力状态、版本日期或核验日期；
- 法规条号重复、顺序错误、首末条与核验清单不一致；
- 执法应用页引用了不存在的法规 ID 或条款号；
- 法条卡文本与法规库同版本条文不一致；
- 三级页没有首屏速览，或将禁止事项、适用条件、法条原文置于默认折叠区；
- 同一页面同时存在旧 `.source-list` 与新公开来源组件；
- 搜索索引不包含新增法规正文或产生重复页面记录。

## 12. 测试与验收

### 12.1 自动化测试

至少增加以下检查：

1. 遍历生产 HTML，断言不存在“巡防百科知识库内部资料”、内部 `.md` 文件名和本地绝对路径。
2. 每个来源项都包含 `http` 或 `https` 链接、标题、发布方和核验日期。
3. 每个学习型三级页包含 `.quick-read`、页内导航和唯一的公开来源区。
4. 所有默认折叠章节使用原生 `<details>`，且禁止事项、适用条件、法律依据不在闭合的 `<details>` 内。
5. URL hash 指向折叠章节时，该章节会自动展开；键盘可操作全部交互。
6. 每部法规页面的条文数量、第一条、最后一条和顺序与 `legal-documents.json` 一致。
7. 每张 `.legal-basis-card` 的条文文本与法规数据中的对应条文完全一致。
8. `fagui/index.html` 的每个“法规全文”入口都指向完整正文页；索引类页面明确标为索引。
9. 搜索“第九条 + 人民警察法”等组合能返回正确法规页及条文锚点。
10. 运行现有全部单元测试，确保旧 URL、导航、鉴权和搜索无回归。

### 12.2 视觉与人工验收

至少选取以下代表页做 1440px 桌面和 390px 移动端截图：

- 一个长警情页；
- 一个装备页；
- 一个执法程序页；
- 一部短法规全文；
- 一部长法规全文。

人工验收标准：

- 首屏不滚动即可看到标题、核心结论和至少两个行动重点。
- 用户不展开任何细节，也能理解核心流程和禁止边界。
- 展开细节后不丢失当前位置，目录高亮正确。
- 法规全文默认完整可见，可直接查找并复制任一条。
- 移动端无横向滚动，目录和长 URL 不撑破容器。
- 打印法规时正文完整，不打印网站导航和折叠控制。
- 深色与浅色主题下重点卡片、法条和来源链接对比度合格。

## 13. 实施顺序

1. 建立测试基线，新增“内部来源不得公开”的失败测试。
2. 统一公开来源数据和渲染器，先移除所有页面中的内部来源展示。
3. 完成学习型三级页组件、交互和一个代表页面迁移，验证桌面/移动/打印行为。
4. 批量迁移其余学习型三级页，并逐页编辑首屏速览内容，禁止用机械截取正文首句代替人工提炼。
5. 建立 `legal-documents.json`、法规生成器和完整性测试。
6. 按官方现行版本逐部录入并核验法规全文。
7. 从统一法规数据生成 5 个执法应用页的法条卡。
8. 重建搜索索引，检查旧 URL、锚点和相关内容链接。
9. 运行全量测试并完成代表页面视觉验收。

## 14. 发布门槛

只有同时满足以下条件才可发布：

- 公开 HTML 中内部来源暴露数量为 0。
- 所有三级页公开来源覆盖率为 100%，且没有空来源卡片。
- 5 个执法应用页涉及的核心法条均在当前页完整展示。
- 法律法规库的单部法规入口均为现行完整正文；合集入口明确标为索引。
- 2025 年修订、2026-01-01 施行的《治安管理处罚法》已替换旧版本内容。
- 法规完整性、条文复用一致性、搜索、链接、响应式和现有回归测试全部通过。

## 15. 给 Claude Code 的执行要求

- 不改变现有公开 URL；确需新增法规页时补充 `data/legacy-url-map.json` 或相关链接检查配置。
- 先写测试再迁移批量页面，避免仅靠人工抽查。
- 不联网自动覆盖法规正文；任何法规版本变更必须留下来源 URL、核验日期和变更记录。
- 不以 JavaScript 动态加载法规正文，正文必须存在于最终静态 HTML。
- 不把所有段落都包成卡片；首屏只突出真正影响判断和行动的信息。
- 不删除内部溯源数据，只禁止其进入公开展示层。
- 不把“官方全文入口”当作完成法律库全文建设的替代品。
